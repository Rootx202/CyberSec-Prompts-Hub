## Prompt 1: Alert Triage and Prioritization

### Description
A systematic approach to triaging and prioritizing security alerts based on severity, impact, and threat intelligence context. This prompt helps SOC analysts efficiently categorize and respond to the high volume of alerts generated by security tools.

### Tags
`alert-triage` `prioritization` `incident-response` `soc-workflow` `threat-intelligence`

---

## 🇬🇧 English Prompt

```
You are a senior SOC analyst responsible for alert triage and prioritization. Analyze the following security alert and provide a comprehensive triage assessment.

ALERT DETAILS:
- Alert ID: [ALERT_ID]
- Source System: [SIEM/EDR/IDS/FIREWALL]
- Timestamp: [TIMESTAMP]
- Alert Type: [ALERT_TYPE]
- Affected Asset(s): [HOSTNAME/IP]
- User Account(s): [USERNAME]
- Raw Log Data: [LOG_DATA]
- MITRE ATT&CK Mapping: [TECHNIQUE_ID]

Perform the following triage tasks:

1. SEVERITY ASSESSMENT
   - Evaluate the potential impact on business operations
   - Assess data sensitivity of affected assets
   - Determine exploitability and threat actor capability
   - Consider current threat landscape and active campaigns

2. PRIORITY CLASSIFICATION
   Assign a priority level (Critical/High/Medium/Low) based on:
   - Business impact potential
   - Asset criticality (Crown Jewels assessment)
   - Threat intelligence correlation
   - Historical false positive rate for this alert type

3. INITIAL CONTEXT ENRICHMENT
   - Correlate with previous alerts (last 24-72 hours)
   - Check threat intelligence feeds for IOC matches
   - Verify asset ownership and business function
   - Identify related user account risk score

4. RECOMMENDED ACTIONS
   - Immediate containment requirements (if any)
   - Investigation steps priority order
   - Stakeholder notification requirements
   - Escalation triggers and thresholds

5. FALSE POSITIVE INDICATORS
   List indicators that might suggest this is a false positive:
   - Known benign activity patterns
   - Scheduled maintenance windows
   - Authorized penetration testing activities
   - Business-justified exceptions

OUTPUT FORMAT:
Provide a structured triage report with clear recommendations and decision rationale.
```

## Prompt 2: Escalation Procedures

### Description
A structured framework for escalating security incidents through appropriate channels based on severity, impact, and organizational requirements. This prompt ensures timely notification of stakeholders and proper escalation paths.

### Tags
`escalation` `incident-management` `communication` `stakeholder-notification` `severity-levels`

---

## 🇬🇧 English Prompt

```
You are a SOC Team Lead managing incident escalation procedures. Develop a comprehensive escalation plan for the following security incident scenario.

INCIDENT SCENARIO:
- Incident ID: [INCIDENT_ID]
- Initial Detection Time: [DETECTION_TIME]
- Incident Category: [CATEGORY - e.g., Ransomware, Data Breach, DDoS, Insider Threat]
- Current Status: [STATUS]
- Affected Systems: [SYSTEMS_LIST]
- Business Units Impacted: [BUSINESS_UNITS]
- Current Containment Status: [CONTAINMENT_STATUS]

Create a detailed escalation plan addressing:

1. ESCALATION MATRIX
   Define escalation paths for each severity level:
   - P1 (Critical): CISO, Legal, PR, Executive Leadership
   - P2 (High): Security Manager, IT Director, Business Unit Head
   - P3 (Medium): SOC Manager, System Owners
   - P4 (Low): SOC Analyst, Asset Owner
   
   Include contact methods and response time SLAs.

2. NOTIFICATION TIMELINE
   - Initial notification (within X minutes of detection)
   - Status update intervals by severity
   - Escalation triggers and timing
   - Regulatory notification requirements (GDPR, HIPAA, etc.)

3. STAKEHOLDER COMMUNICATION TEMPLATES
   Provide templates for:
   - Executive Summary (C-Suite level)
   - Technical Brief (IT Leadership)
   - Operational Update (SOC Team)
   - External Communication (if required)

4. ESCALATION DECISION CRITERIA
   Define clear criteria for escalating:
   - Incident scope expansion thresholds
   - Business impact escalation triggers
   - Regulatory/compliance escalation requirements
   - Media/public relations triggers

5. DE-ESCALATION CRITERIA
   When and how to de-escalate:
   - False positive confirmation
   - Successful containment without impact
   - Incident scope reduction
   - Transfer to lower-tier support

6. DOCUMENTATION REQUIREMENTS
   - Incident ticket updates
   - Communication logs
   - Decision rationale documentation
   - Lessons learned capture

OUTPUT FORMAT:
Provide a comprehensive escalation playbook with clear procedures and templates.
```

### Tags

```

`metrics` `kpi` `performance-measurement` `reporting` `soc-effectiveness`

- Industry: [INDUSTRY_TYPE]
